CHAPITRE 2 - CONCEPTS PTHREADS FONDAMENTAUX
================================================================================

[ ] 2.1 pthread_mutex_t
    [ ] 2.1.1 Définition : verrou d'exclusion mutuelle (une seule section
              critique à la fois)
    [ ] 2.1.2 Cycle de vie : pthread_mutex_init -> lock/unlock -> destroy
    [ ] 2.1.3 Que se passe-t-il si deux threads locke le même mutex en même
              temps ? -> le second est BLOQUÉ jusqu'à unlock du premier
    [ ] 2.1.4 Donner les 4 mutex du projet et la donnée que chacun protège :
              - log_mutex        : sérialise les printf (une ligne complète par printf)
              - stop_mutex       : protège sim->stop et ticket_counter
              - coder_state_mutex: protège last_compile_start_ms et compiles_done
              - dongle[i].mutex  : protège taken, released_at_ms, et la heap waiters

[ ] 2.2 pthread_cond_t
    [ ] 2.2.1 Définition : variable de condition, permet à un thread d'attendre
              qu'une condition soit vraie sans faire de polling actif
    [ ] 2.2.2 Cycle de vie : pthread_cond_init -> wait/broadcast -> destroy
    [ ] 2.2.3 Règle fondamentale : cond_wait libère ATOMIQUEMENT le mutex
              et se met en attente. À la sortie, il re-lock le mutex.
              NE PAS unlock avant le cond_wait -> double-unlock -> UB
    [ ] 2.2.4 Pourquoi toujours while(!condition) et jamais if(!condition) ?
              - Spurious wakeup : POSIX autorise un réveil sans raison
              - Broadcast collectif : tous se réveillent, seul un peut prendre,
                les autres doivent se rendormir
    [ ] 2.2.5 pthread_cond_broadcast vs pthread_cond_signal :
              - signal  : réveille UN thread (indéterminé)
              - broadcast: réveille TOUS les threads en attente sur la cond
              -> le sujet impose broadcast (pas de signal) : conséquence directe
                 sur la boucle while qui re-teste la condition

[ ] 2.3 pthread_cond_timedwait
    [ ] 2.3.1 Différence avec cond_wait : timeout absolu (struct timespec)
    [ ] 2.3.2 Pourquoi l'utiliser ici ? Cooldown : on sait exactement quand
              le cooldown expire -> on dort exactement le temps restant
    [ ] 2.3.3 Pourquoi une timespec ABSOLUE et pas un delta ?
              -> pthread_cond_timedwait prend un instant absolu, pas une durée

[ ] 2.4 Data race
    [ ] 2.4.1 Définition : deux threads accèdent à la même variable en même
              temps, dont au moins une écriture, sans synchronisation
    [ ] 2.4.2 Conséquence : comportement indéfini (UB) – résultat imprévisible
    [ ] 2.4.3 Exemple dans le projet : sim->stop lu par coders et écrit par
              monitor -> protégé par stop_mutex
    [ ] 2.4.4 Torn read : sur 64 bits, lire une long long peut nécessiter deux
              instructions 32 bits -> lecture d'une valeur à moitié ancienne.
              C'est pourquoi last_compile_start_ms est lu sous coder_state_mutex

CHAPITRE 3 - DEADLOCK ET STRATÉGIE ANTI-DEADLOCK
================================================================================

[ ] 3.1 Les 4 conditions de Coffman (toutes nécessaires pour un deadlock)
    [ ] 3.1.1 Exclusion mutuelle  : une ressource ne peut être tenue que par
              un thread à la fois
    [ ] 3.1.2 Hold and wait       : un thread tient une ressource et attend d'en
              obtenir une autre
    [ ] 3.1.3 No preemption       : on ne peut pas forcer un thread à lâcher
              une ressource
    [ ] 3.1.4 Attente circulaire  : A attend B, B attend C, C attend A

[ ] 3.2 Stratégie choisie : Option B (asymétrie pair/impair)
    [ ] 3.2.1 Comment ça marche :
              - coder pair (0, 2, 4...) : gauche d'abord, puis droite
              - coder impair (1, 3, 5...) : droite d'abord, puis gauche
    [ ] 3.2.2 Quelle condition de Coffman est cassée ?
              -> L'ATTENTE CIRCULAIRE : avec N coders en cercle, si le dernier
              prend droite en premier (impair), il ne complète jamais le cercle
              d'attente -> deadlock impossible
    [ ] 3.2.3 Savoir l'illustrer avec 5 coders et 5 dongles (schéma en cercle)
    [ ] 3.2.4 Alternatives non choisies :
              - Option A : ordre par adresse mémoire (même principe, ordre global)
              - Option C : arbitrator mutex global (sérialise toutes les prises)
    [ ] 3.2.5 Libération en ordre LIFO (inverse de l'acquisition) :
              - pair acquiert gauche puis droite -> libère droite puis gauche
              - minimise les inversions de priorité dans la heap

[ ] 3.3 Cas particulier : 1 seul coder
    [ ] 3.3.1 N=1 -> 1 seul dongle sur la table
    [ ] 3.3.2 Le coder a besoin de 2 dongles pour compiler, il n'en a qu'un
              -> il ne peut jamais compiler -> il burnout
    [ ] 3.3.3 C'est le comportement ATTENDU selon le sujet, pas un bug

CHAPITRE 4 - STARVATION ET POLITIQUES DE SCHEDULING
================================================================================

[ ] 4.1 Définition de la starvation
    [ ] 4.1.1 Un thread reste indéfiniment bloqué car d'autres ont toujours
              la priorité -> il ne progresse jamais
    [ ] 4.1.2 Différence avec le deadlock : en starvation il n'y a pas de
              cycle d'attente, le thread éligible est juste systématiquement
              doublé par d'autres

[ ] 4.2 Politique FIFO (First In, First Out)
    [ ] 4.2.1 Mécanisme : ticket_counter global, incrémenté sous stop_mutex
              pour chaque demande. La clé dans la heap = ce ticket.
    [ ] 4.2.2 Propriété : le coder qui arrive en premier sur le dongle repart
              en premier -> ordre d'arrivée strict
    [ ] 4.2.3 Garantit-il la liveness ? OUI : chaque coder finit par être
              le plus ancien -> il passe en tête -> il obtient le dongle

[ ] 4.3 Politique EDF (Earliest Deadline First)
    [ ] 4.3.1 Mécanisme : deadline = last_compile_start_ms + time_to_burnout
              La clé dans la heap = cette deadline
    [ ] 4.3.2 Intuition : celui qui va brûler le plus tôt passe en premier
    [ ] 4.3.3 Calcul de la clé AVANT de locker le dongle.mutex :
              Pourquoi ? Car on lit coder_state_mutex pour last_compile_start_ms.
              L'ordre de lock est stop/coder_state AVANT dongle.mutex.
              Si on était déjà sous dongle.mutex on ne pourrait pas prendre
              coder_state_mutex sans violer l'ordre.

[ ] 4.4 Le heap est "agnostique" au scheduler
    [ ] 4.4.1 La heap ne sait pas si la clé est un ticket FIFO ou une deadline
    [ ] 4.4.2 C'est l'appelant (dongle_compute_key) qui calcule la bonne clé
    [ ] 4.4.3 Avantage : un seul heap générique, pas de code dupliqué

CHAPITRE 5 - HEAP BINAIRE (PRIORITY QUEUE)
================================================================================

[ ] 5.1 Définition et propriétés d'un heap min
    [ ] 5.1.1 Arbre binaire quasi-complet stocké dans un tableau
    [ ] 5.1.2 Propriété de heap : parent <= ses deux enfants (pour un min-heap)
    [ ] 5.1.3 Le minimum est toujours à l'indice 0 (la racine)

[ ] 5.2 Formules d'indexation dans le tableau
    [ ] 5.2.1 Parent de i        : (i - 1) / 2
    [ ] 5.2.2 Enfant gauche de i : 2 * i + 1
    [ ] 5.2.3 Enfant droit de i  : 2 * i + 2

[ ] 5.3 Opérations et complexités
    [ ] 5.3.1 pq_push   : O(log n) — ajouter en fin + sift_up
    [ ] 5.3.2 pq_pop    : O(log n) — sauver racine, mettre dernier en tête + sift_down
    [ ] 5.3.3 pq_peek   : O(1)     — lire data[0] sans modifier
    [ ] 5.3.4 pq_remove : O(n)     — recherche linéaire, puis sift_up ou sift_down
    [ ] 5.3.5 pq_grow   : amortized O(1) — double la capacité quand pleine

[ ] 5.4 sift_up (remonter)
    [ ] 5.4.1 Contexte : après un push, le nouveau noeud est en fin
    [ ] 5.4.2 On compare avec le parent ; si on est plus petit on échange
    [ ] 5.4.3 On continue jusqu'à racine ou jusqu'à ce que la propriété soit OK

[ ] 5.5 sift_down (descendre)
    [ ] 5.5.1 Contexte : après un pop (la racine disparaît, on met le dernier à sa place)
    [ ] 5.5.2 On compare avec le plus petit des deux enfants
    [ ] 5.5.3 Si l'enfant est plus petit, on échange, on continue vers le bas

[ ] 5.6 pq_remove (retrait en milieu de heap)
    [ ] 5.6.1 Trouver l'élément par recherche linéaire (O(n))
    [ ] 5.6.2 Remplacer par le dernier élément du tableau
    [ ] 5.6.3 Le nouveau noeud peut monter (sift_up) ou descendre (sift_down)
              -> on teste les deux : on compare avec le parent pour décider

[ ] 5.7 Tiebreak
    [ ] 5.7.1 Deux coders peuvent avoir exactement la même clé EDF
              (ex: même deadline) ou FIFO si ticket_counter utilisé deux fois
    [ ] 5.7.2 On départage par coder_id (unique) pour avoir un ordre déterministe
    [ ] 5.7.3 Sans tiebreak : sift_down avec "<=" peut créer des swaps infinis

[ ] 5.8 Dynamisme du heap
    [ ] 5.8.1 Capacité initiale = 4 (pour chaque dongle) — au plus N waiters
    [ ] 5.8.2 pq_grow alloue un nouveau tableau 2x plus grand, recopie, free l'ancien
    [ ] 5.8.3 Pourquoi pas realloc ? Certains sujets 42 l'interdisent,
              et la recopie manuelle est sûre (pas d'alignement ambiguë)

CHAPITRE 6 - GESTION DU TEMPS
================================================================================

[ ] 6.1 now_ms() — source de temps
    [ ] 6.1.1 Utilise gettimeofday (POSIX) : précision à la microseconde
    [ ] 6.1.2 Retourne ms depuis l'epoch UNIX (long long, ~1.7e12)
    [ ] 6.1.3 Pourquoi long long ? Un int 32 bits ne peut pas contenir 1.7e12
    [ ] 6.1.4 Pourquoi 1000LL ? Force la multiplication en 64 bits (évite overflow)

[ ] 6.2 elapsed_ms(sim) — timestamps relatifs dans les logs
    [ ] 6.2.1 Retourne now_ms() - sim->start_ms
    [ ] 6.2.2 Affiche "200 1 is compiling" au lieu d'un epoch absolu

[ ] 6.3 precise_sleep_ms() — sleep précis et interruptible
    [ ] 6.3.1 Pourquoi pas usleep(ms * 1000) directement ?
              -> usleep peut dormir PLUS que demandé (granularité du scheduler OS)
    [ ] 6.3.2 Boucle : usleep(100 µs) + vérification horloge réelle
    [ ] 6.3.3 Lit sim->stop SOUS stop_mutex à chaque itération pour sortir vite
    [ ] 6.3.4 Impact helgrind : sans le mutex, helgrind détectait un data race

[ ] 6.4 ms_to_timespec() — pour pthread_cond_timedwait
    [ ] 6.4.1 pthread_cond_timedwait attend un INSTANT ABSOLU (struct timespec)
              pas une durée
    [ ] 6.4.2 ts.tv_nsec = (ms % 1000) * 1000000LL -> LL obligatoire
              (999 * 1000000 = 999 000 000 > INT_MAX si en 32 bits)

CHAPITRE 7 - LE THREAD MONITOR
================================================================================

[ ] 7.1 Rôle du monitor
    [ ] 7.1.1 Surveille TOUS les coders en parallèle dans une boucle
    [ ] 7.1.2 Deux conditions de sortie :
              - burnout : un coder n'a pas compilé depuis time_to_burnout
              - fin normale : tous les coders ont atteint leur objectif

[ ] 7.2 Précision < 10 ms (exigée par le sujet)
    [ ] 7.2.1 usleep(500) = 0.5 ms entre chaque tour de boucle
    [ ] 7.2.2 Au pire : burnout réel à T, monitor le détecte à T + 0.5 ms
              -> bien sous la limite de 10 ms

[ ] 7.3 monitor_check_burnout
    [ ] 7.3.1 Lit last_compile_start_ms et compiles_done SOUS coder_state_mutex
    [ ] 7.3.2 Ignore les coders qui ont fini (compiles_done >= required) :
              leur last_compile_start_ms est figé, ne pas les compter comme
              brûlés alors qu'ils ont terminé normalement
    [ ] 7.3.3 Calcule since = now_ms() - last HORS du mutex (syscall hors zone critique)

[ ] 7.4 monitor_trigger_burnout
    [ ] 7.4.1 Appelle log_burned_out (qui set stop=1 sous log_mutex, atomiquement)
    [ ] 7.4.2 Puis broadcast sur TOUS les dongles pour réveiller les coders
              bloqués en cond_timedwait -> ils voient stop=1 et sortent

[ ] 7.5 Pourquoi le monitor est lancé EN PREMIER (avant les coders) ?
    [ ] 7.5.1 Pour qu'il surveille dès le départ
    [ ] 7.5.2 last_compile_start_ms est initialisé à sim->start_ms dans main
              AVANT pthread_create(monitor) -> pas de faux burnout dès la 1ère ms

[ ] 7.6 monitor_signal_all_dongles
    [ ] 7.6.1 Lock/broadcast/unlock sur chaque dongle
    [ ] 7.6.2 Pourquoi lock avant broadcast ?
              pthread_cond_broadcast doit être appelé sous le même mutex que
              cond_wait pour garantir que le signal n'est pas perdu
              (race : thread s'endort juste après que stop=1 -> sans broadcast
               il dormirait jusqu'au timeout)

CHAPITRE 8 - ORDRE DES LOCKS ET PRÉVENTION DES INTERBLOCAGES DE MUTEX
================================================================================

[ ] 8.1 L'ordre de lock global du projet
    Règle : toujours prendre les mutex dans cet ordre, JAMAIS dans l'ordre inverse.
    log_mutex -> stop_mutex -> coder_state_mutex -> dongle[i].mutex

[ ] 8.2 Pourquoi un ordre total évite le deadlock entre mutex ?
    [ ] 8.2.1 Avec un ordre, on casse la condition d'attente circulaire sur les mutex eux-mêmes
    [ ] 8.2.2 Exemple de violation : si on prenait dongle.mutex PUIS stop_mutex,
              le monitor (qui prend stop puis dongle) et un coder (dongle puis stop)
              pourraient se bloquer mutuellement

[ ] 8.3 Cas particulier : dongle_wait_loop lit sim->stop SANS stop_mutex
    [ ] 8.3.1 Pourquoi ? On est déjà sous dongle.mutex, stop_mutex vient avant
              dans l'ordre -> prendre stop_mutex ici violerait l'ordre
    [ ] 8.3.2 Accepté car : le monitor broadcast sur le dongle après avoir mis stop=1,
              donc même si on lit stop un peu tard (race bénigne), on sortira de
              la boucle au prochain tour (au max 2ms après le timedwait)

[ ] 8.4 log_burned_out est atomique du point de vue de stop
    [ ] 8.4.1 Set sim->stop = 1 SOUS log_mutex (cf. log_burnout.c)
    [ ] 8.4.2 Le log "burned out" et stop=1 ne peuvent pas être séparés par
              un autre log -> garantit qu'aucun log n'apparaît après "burned out"

CHAPITRE 9 - CYCLE DE VIE D'UN CODER
================================================================================

[ ] 9.1 Séquence complète d'un cycle
    [ ] 9.1.1 coder_take_dongles : acquiert gauche et droite (ordre selon parité)
    [ ] 9.1.2 MAJ last_compile_start_ms SOUS coder_state_mutex
    [ ] 9.1.3 log "is compiling" + precise_sleep_ms(time_to_compile)
    [ ] 9.1.4 compiles_done++ SOUS coder_state_mutex
    [ ] 9.1.5 coder_release_dongles (ordre LIFO)
    [ ] 9.1.6 log "is debugging" + sleep
    [ ] 9.1.7 log "is refactoring" + sleep
    [ ] 9.1.8 Retour au test coder_should_continue

[ ] 9.2 Pourquoi les dongles sont relâchés AVANT debug/refactor ?
    [ ] 9.2.1 Debug et refactor ne nécessitent pas la ressource
    [ ] 9.2.2 Libérer tôt maximise le throughput : d'autres coders peuvent
              compiler pendant qu'on débugge

[ ] 9.3 Initialisation de last_compile_start_ms
    [ ] 9.3.1 Initialisé à sim->start_ms (pas à 0 !) SOUS coder_state_mutex
    [ ] 9.3.2 Si initialisé à 0 : since = now - 0 = ~1.7e12 ms >> time_to_burnout
              -> tous les coders sont immédiatement déclarés en burnout

[ ] 9.4 coder_should_continue
    [ ] 9.4.1 Lit stop SOUS stop_mutex
    [ ] 9.4.2 Lit compiles_done SOUS coder_state_mutex
    [ ] 9.4.3 Retourne 0 (stop) si stop=1 OU si l'objectif est atteint

[ ] 11.1 Pourquoi sérialiser les logs ?
    [ ] 11.1.1 printf n'est pas thread-safe : deux threads peuvent écrire en
              même temps, produisant des lignes entrelacées
    [ ] 11.1.2 log_mutex garantit qu'un seul printf s'exécute à la fois

[ ] 11.2 Format exact des messages (imposé par le sujet)
    [ ] 11.2.1 "%lld %d has taken a dongle\n"
    [ ] 11.2.2 "%lld %d is compiling\n"
    [ ] 11.2.3 "%lld %d is debugging\n"
    [ ] 11.2.4 "%lld %d is refactoring\n"
    [ ] 11.2.5 "%lld %d burned out\n"
    Le premier %lld est le timestamp RELATIF (elapsed_ms, pas epoch absolu)

[ ] 11.3 Garantie "aucun log après burned out"
    [ ] 11.3.1 log_event vérifie sim->stop AVANT de printf
    [ ] 11.3.2 log_burned_out set stop=1 SOUS log_mutex, donc ces deux actions

[ ] 11.4 Pourquoi logger "taken a dongle" APRÈS avoir délocké dongle.mutex ?
    [ ] 11.4.1 log_taken_dongle appelle log_event qui prend log_mutex
    [ ] 11.4.2 L'ordre est : log_mutex AVANT dongle.mutex
    [ ] 11.4.3 Si on loggait sous dongle.mutex : violation de l'ordre de lock
              -> deadlock potentiel

CHAPITRE 14 - PIÈGES ET QUESTIONS PIÈGES DE L'ÉVALUATEUR
================================================================================

[ ] 14.1 "Pourquoi while et pas if pour cond_wait ?"
    -> Spurious wakeup + broadcast collectif (cf. 2.2.4)

[ ] 14.2 "Que se passe-t-il si dongle_cooldown = 0 ?"
    -> La condition (released_at_ms > 0 && elapsed < cooldown) est fausse
       avec cooldown=0 -> le cooldown ne bloque jamais -> comportement normal

[ ] 14.3 "Que se passe-t-il si tous les coders finissent en même temps ?"
    -> monitor_check_end retourne 1 -> stop=1 + broadcast -> propre

[ ] 14.4 "Pourquoi initialiser released_at_ms à 0 et pas à now_ms() ?"
    -> À 0, la condition released_at_ms > 0 est fausse : pas de cooldown
       au démarrage (le dongle n'a jamais été utilisé, donc pas de cooldown
       à respecter). Initialiser à now_ms() créerait un cooldown artificiel
       au lancement.

[ ] 14.5 "Pourquoi le monitor broadcast sur les dongles plutôt que d'utiliser
          une cond_var globale ?"
    -> Les coders attendent sur dongle[i].cond sous dongle[i].mutex.
       Un broadcast sur une cond globale (sans le bon mutex) ne les réveillerait pas.
       Le broadcast doit être fait sous LE MÊME mutex que le cond_wait.

[ ] 14.6 "Comment garantir qu'aucun log n'est émis après burned out ?"
    -> log_event vérifie sim->stop sous log_mutex avant de printf.
       log_burned_out set stop=1 sous log_mutex (atomique).
       Un log concurrent attendra la release du log_mutex et verra stop=1
       -> sortie sans printf.

[ ] 14.7 "Quel est le pire cas de latence du monitor ?"
    -> 0.5 ms (usleep 500) + overhead syscall. Très inférieur à la limite de 10 ms.

[ ] 14.8 "Qu'est-ce qu'un faux positif helgrind pour cond_broadcast ?"
    -> Helgrind signale "dubious broadcast" quand le broadcast se produit sans
       tenir le mutex associé. Dans notre code, le broadcast est TOUJOURS sous
       le mutex (dongle_release, monitor_signal_all_dongles). Helgrind se trompe
       quand un timedwait expire exactement au moment du broadcast (race interne
       à helgrind, pas dans notre code).

[ ] 14.9 "Pourquoi utiliser un heap et pas une simple liste triée ou une queue ?"
    -> Push O(log n) vs O(n) pour une liste triée, pop O(log n) vs O(1).
       Le sujet impose un heap custom -> on ne peut pas utiliser une structure plus
       simple même si elle serait suffisante pour N réduit.

[ ] 14.10 "Que se passe-t-il en EDF quand deux coders ont la même deadline ?"
    -> Le tiebreak (coder_id) départage de façon déterministe.
       Sans tiebreak, un comparateur "<=" créerait des swaps infinis dans sift_down.
